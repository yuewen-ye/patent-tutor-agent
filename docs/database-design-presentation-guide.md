# 专利导学系统数据库设计讲解

## 1. 数据库在系统中的作用

系统使用 MySQL 保存需要长期保留、查询、关联和审计的结构化业务数据，包括：

- 学员身份；
- 课程和反馈会话；
- 当前画像和历史画像；
- 学习路径；
- 模型生成的题目；
- 学员作答；
- 知识点掌握度；
- 工作流状态和事件；
- 课程产物索引；
- 法条引用关系。

MySQL 可以理解为系统的业务账本。它记录发生了什么、数据属于谁、不同数据之间有什么关系，以及
数据为什么发生变化。

---

## 2. 不同类型的数据存放在哪里

| 数据类型 | 存放位置 | 原因 |
|---|---|---|
| 学员、会话、画像、路径、题目、作答、BKT | MySQL | 需要事务、外键、唯一约束、查询和审计 |
| 完整工作流状态 | MySQL `session_states` | 便于查询会话详情和读取最终状态 |
| 课程、专家过程、反馈 Markdown | `artifacts/` | 正文较长，适合文件存储和人工阅读 |
| Artifact 路径、哈希、类型、归属 | MySQL `artifacts` | 便于查询、校验和关联 |
| 法律资料向量 | Milvus | 用于向量相似度检索 |
| 知识 DAG、易混淆对 | 后端静态 JSON | 是全体学员共享的课程定义 |
| 工作流节点间的即时状态 | `StateDict` | 用于一次工作流内部的数据传递 |

数据库不是只保存 `artifacts` 文件路径。文件路径只是其中一部分；学员画像、题目、答案、作答、
掌握度和会话状态都直接保存在 MySQL 中。

---

## 3. 数据库总体划分

当前数据库共有 25 张表，可以按照八个业务区域理解。

| 业务区域 | 表 |
|---|---|
| 数据库版本 | `schema_migrations` |
| 学员身份 | `students`、`auth_sessions` |
| 会话运行 | `sessions`、`session_states`、`session_events`、`session_checkpoints`、`rounds` |
| 学员画像 | `student_profiles`、`profile_history`、`student_weak_points`、`memory_items` |
| 自适应学习 | `student_node_mastery`、`mastery_events`、`learning_paths`、`session_directives` |
| 教学闭环 | `onboarding_responses`、`questions`、`attempts`、`feedback_logs` |
| 产物和引用 | `artifacts`、`legal_citations`、`artifact_citations` |
| 静态课程目录 | `knowledge_nodes`、`confusion_pairs` |

---

## 4. 学员身份相关表

### 4.1 `students`

`students` 是学员根表。画像、会话、作答、掌握度等数据最终都归属于某个学员。

主要保存：

- 学员 ID；
- 登录标识；
- 密码哈希；
- 展示名称；
- 邮箱；
- 账号状态；
- 创建和更新时间。

一个学员可以对应多个课程会话、多次作答、多份画像历史和多个知识点掌握度记录。

### 4.2 `auth_sessions`

`auth_sessions` 用于保存登录会话：

- 登录会话 ID；
- 所属学员；
- 令牌哈希；
- 过期时间；
- 撤销时间；
- 创建时间。

令牌只保存哈希，不保存明文。

---

## 5. 会话运行相关表

### 5.1 `sessions`

`sessions` 是工作流会话主表。课程生成、诊断、聊天和反馈都使用同一张会话表，通过
`workflow_mode` 区分类型。

主要保存：

- 会话 ID；
- 所属学员；
- 父会话 ID；
- 工作流类型；
- 会话状态；
- 学习目标；
- 初始输入；
- 失败原因；
- 工作流版本；
- 创建、更新和完成时间。

课程会话和反馈会话是两条不同记录。反馈会话通过 `parent_session_id` 指向原课程会话。

### 5.2 `session_states`

`session_states` 保存每个会话最新的完整结构化状态。

主要内容包括：

- 学员画像；
- 学习路径；
- 专家生成结果；
- 最终课程包；
- Judge 结果；
- 反馈结果；
- Artifact 列表；
- 当前工作流字段。

一个会话只对应一条当前状态记录。`revision` 用于防止旧状态覆盖新状态。

### 5.3 `session_events`

`session_events` 保存工作流执行过程中产生的事件，例如：

- 某个节点开始；
- 某个节点完成；
- 会话失败；
- 会话取消；
- 某个阶段发生变化。

每个事件都有会话内序号，因此可以按照固定顺序还原执行过程。

### 5.4 `rounds`

`rounds` 保存专家协作轮次和审核过程。

主要保存：

- 所属课程会话；
- 轮次编号；
- 整合尝试次数；
- 当前阶段；
- 轮次状态；
- Judge 决策；
- 开始和完成时间。

当课程需要重新整合时，系统增加新的 `integration_attempt`，不会覆盖之前的整合结果。

### 5.5 `session_checkpoints`

`session_checkpoints` 用于保存工作流 checkpoint，包括：

- checkpoint ID；
- 业务会话 ID；
- LangGraph thread ID；
- checkpoint 内容；
- checkpoint 元数据；
- 创建时间。

它与 `session_states` 的区别是：

- `session_states` 保存业务上最新的完整状态；
- `session_checkpoints` 用于从工作流中断位置继续执行。

---

## 6. 学员画像相关表

### 6.1 `student_profiles`

`student_profiles` 保存学员当前最新画像，一名学员只有一条。

主要保存：

- 完整画像 JSON；
- 当前知识水平；
- 画像版本；
- 最近更新时间。

当系统需要快速获取学员当前状态时，直接读取这张表。

### 6.2 `profile_history`

`profile_history` 保存画像历史快照。

每条记录包含：

- 所属学员；
- 产生画像的会话；
- 产生画像的轮次；
- 画像来源；
- 画像版本；
- 当时的完整画像；
- 当时的掌握度快照；
- 快照时间。

`student_profiles` 表示“现在是什么样”，`profile_history` 表示“过去每个阶段是什么样”。

### 6.3 `student_weak_points`

`student_weak_points` 把画像中的薄弱点拆成可查询记录。

主要保存：

- 薄弱点描述；
- 对应知识节点；
- 来源阶段；
- 当前状态；
- 首次出现时间；
- 最近出现时间。

薄弱点状态可以是有效、已解决或已被新记录替代。

### 6.4 `memory_items`

`memory_items` 保存 Agent 使用的历史记忆和上下文。

它用于记录学习事件和可回忆内容，不是当前画像或当前掌握度的权威来源：

- 当前画像以 `student_profiles` 为准；
- 当前掌握度以 `student_node_mastery` 为准；
- `memory_items` 只提供补充上下文。

---

## 7. 自适应学习相关表

### 7.1 `student_node_mastery`

`student_node_mastery` 保存学员在每个知识节点上的当前 BKT 掌握度。

一条记录对应：

> 一个学员 + 一个知识节点。

主要保存：

- 当前掌握概率 P(L)；
- 观测次数；
- 正确次数；
- 错误次数；
- 最近一次作答；
- BKT 模型版本；
- 更新时间。

Planner 使用这里的当前值调整学习路径和题目难度。

### 7.2 `mastery_events`

`mastery_events` 保存每一次掌握度变化的原因和过程。

主要保存：

- 哪名学员；
- 哪个知识节点；
- 哪次作答触发；
- 作答正确还是错误；
- 更新前 P(L)；
- 后验 P(L)；
- 最终 P(L)；
- 本次使用的 BKT 参数；
- BKT 模型版本；
- 发生时间。

`student_node_mastery` 保存当前结果，`mastery_events` 解释当前结果是怎样得到的。

### 7.3 `learning_paths`

`learning_paths` 保存一次课程规划得到的知识节点顺序。

主要保存：

- 所属会话；
- 路径版本；
- 知识节点；
- 节点名称；
- 前置知识；
- 题目难度上限；
- 教学策略；
- 路径顺序。

重新规划时增加新的路径版本，不覆盖旧路径。

### 7.4 `session_directives`

`session_directives` 保存 Planner 下发给课程生成阶段的教学指令。

主要包括：

- 出题范围；
- 向后复习范围；
- 向前探测范围；
- 薄弱点探测范围；
- 降维、进阶或薄弱点跟进指令；
- 指令版本。

---

## 8. 教学闭环相关表

### 8.1 `onboarding_responses`

`onboarding_responses` 保存学员提交的原始问卷答案。

主要保存：

- 学员；
- 由问卷创建的课程会话；
- 问卷版本；
- 原始回答 JSON；
- 提交时间。

题目正文和选项来自版本化问卷定义。为了让诊断阶段理解答案，系统还会把题目、选项、所选项和
开放回答一起放入课程会话的 `input_payload`。

### 8.2 `questions`

`questions` 保存课程生成过程中产生的题目实例。

这些题目不是从固定题库中读取，而是根据学员画像、学习路径和知识点动态生成。

主要保存：

- 所属课程会话；
- 所属专家轮次；
- 课程内题号；
- 题目类型；
- 认知类别；
- 难度；
- 知识节点；
- 题干；
- 选项；
- 内部参考答案；
- 参考证据；
- 题目版本；
- 发布状态。

题目实例保存后，后续作答可以准确关联到当时生成的题目和参考答案。

### 8.3 `attempts`

`attempts` 保存学员对题目的实际作答。

主要保存：

- 学员；
- 题目实例；
- 所属反馈会话；
- 原始答案；
- 所选选项；
- 正误结果；
- 判题状态；
- 判题来源；
- 作答耗时；
- 幂等键；
- 提交和判题时间。

`is_correct` 可以为空，因为并不是所有答案都能立即得到可靠的二值判断。

唯一幂等键用于防止同一次答案被重复提交和重复计入 BKT。

### 8.4 `feedback_logs`

`feedback_logs` 保存一次反馈阶段的结构化摘要。

主要保存：

- 所属学员；
- 反馈会话；
- 对应画像历史；
- 评价信号；
- BKT 更新摘要；
- 创建时间。

详细反馈正文保存在 Artifact 文件中，结构化摘要保存在数据库中。

---

## 9. 产物和法条引用相关表

### 9.1 `artifacts`

`artifacts` 保存 Markdown 文件的索引，不保存完整 Markdown 正文。

主要保存：

- Artifact ID；
- 所属会话；
- 所属轮次；
- 产物类型；
- 对应状态字段；
- 文件路径；
- SHA-256；
- 创建者；
- 标题；
- 创建时间。

数据库可以通过路径找到文件，通过 SHA-256 检查文件是否被修改，通过会话和轮次确定文件归属。

### 9.2 `legal_citations`

`legal_citations` 保存课程使用的法条和来源信息。

主要保存：

- 法条；
- 来源名称；
- 来源地址；
- 检索片段；
- 检索方式；
- 必要引文；
- 核验状态；
- 创建时间。

核验状态区分未核验、已核验和已拒绝。

### 9.3 `artifact_citations`

`artifact_citations` 是 Artifact 和法条引用之间的关联表。

它表达：

- 一份课程产物可以使用多条法条；
- 同一条法条可以被多份课程产物使用；
- 同一条引用可以在一份产物的不同位置出现。

## 10. 数据库版本和静态目录表

### 10.1 `schema_migrations`

`schema_migrations` 保存已经执行过的数据库结构升级记录。

每条记录只有：

- 迁移版本；
- 执行时间。

它不保存业务数据，也不保存完整 SQL。应用通过这张表判断哪些数据库结构升级已经执行、哪些仍需
执行。

### 10.2 `knowledge_nodes`

`knowledge_nodes` 用于保存带版本的知识 DAG 节点，包括：

- 目录版本；
- 节点 ID；
- 节点名称；
- 前置节点；
- 难度提示；
- 来源路径；
- 是否有效。

### 10.3 `confusion_pairs`

`confusion_pairs` 用于保存带版本的易混淆概念对，包括：

- 目录版本；
- 混淆对 ID；
- 两个概念；
- 标题；
- 混淆原因；
- 关联知识节点。

---

## 11. 核心表关系

### 11.1 学员与画像

- 一个学员对应一份当前画像；
- 一个学员对应多份历史画像；
- 一个学员对应多个薄弱点；
- 一个学员在每个知识节点上对应一条当前 mastery。

### 11.2 学员与会话

- 一个学员可以有多个课程、诊断、聊天和反馈会话；
- 一个反馈会话通过父会话 ID 指向原课程会话；
- 一个会话对应一条当前状态；
- 一个会话对应多个事件和多个专家轮次。

### 11.3 课程、题目和作答

- 一个课程会话可以生成多道题目；
- 一道题目可以对应多次学员作答；
- 每次作答属于一个反馈会话；
- 作答通过题目关联到原课程会话；
- 一次已判定作答最多对应一次 mastery 更新事件。

### 11.4 Artifact 和法条

- 一个会话可以产生多个 Artifact；
- Artifact 可以属于某个专家轮次，也可以只属于会话；
- Artifact 和法条引用是多对多关系。

## 12. 完整业务流程中的数据库读写

### 12.1 问卷提交

写入：

- 学员身份；
- 课程会话；
- 原始问卷回答；
- 初始会话状态；
- 问卷相关 Artifact 索引。

### 12.2 初始诊断

读取：

- 问卷完整上下文；
- 历史画像和记忆。

写入：

- 当前画像；
- 画像历史；
- 薄弱点；
- 会话状态和事件；
- 画像 Artifact 索引。

### 12.3 路径规划

读取：

- 当前画像；
- 当前 mastery；
- 静态知识 DAG 和易混淆对。

写入：

- 学习路径；
- 教学指令；
- 会话状态和事件；
- 路径 Artifact 索引。

### 12.4 课程生成

写入：

- 专家轮次；
- 会话状态和事件；
- 模型生成题目；
- 课程和过程 Artifact 索引；
- 法条引用及其与 Artifact 的关系。

### 12.5 学员作答

读取：

- 原课程会话；
- 题目和内部参考答案；
- 已有作答；
- 当前 mastery。

写入：

- 独立反馈会话；
- 学员原始作答；
- 判题状态和结果；
- 更新后的 mastery；
- mastery 更新事件。

作答、当前 mastery 和 mastery 事件在同一事务中完成。

### 12.6 反馈更新

读取：

- 学员原始作答；
- 判题结果；
- 当前画像；
- 课程上下文。

写入：

- 反馈日志；
- 新画像历史；
- 最新画像；
- 最新薄弱点；
- 反馈会话状态和事件；
- 反馈 Artifact 索引。

## 13. 关键设计理由

### 13.1 当前值与历史记录分开

| 当前值 | 历史依据 |
|---|---|
| `student_profiles` | `profile_history` |
| `student_node_mastery` | `attempts`、`mastery_events` |
| `session_states` | `session_events`、`rounds` |

当前值用于快速查询，历史记录用于解释变化原因。

### 13.2 课程会话与反馈会话分开

课程生成结束后，学员可能过一段时间才作答。独立反馈会话能够：

- 表达课程与反馈的时间边界；
- 保存一次独立反馈过程；
- 允许同一课程后续产生多次反馈；
- 避免课程生成工作流长时间等待。

### 13.3 题目作为独立实例保存

题目由课程动态生成，因此每次课程中的题目都是独立实例。保存题目实例可以确保：

- 作答关联到正确的题目版本；
- 参考答案不会因为后续课程变化而丢失；
- 可以按照知识点、认知类别和难度分析作答；
- 可以追溯题目由哪个会话和轮次产生。

### 13.4 作答和 BKT 更新保持事务一致

作答记录、当前 mastery 和 mastery 事件必须同时成功或同时失败，避免数据只更新一部分。

### 13.5 状态快照与事件记录分开

状态快照回答“现在是什么状态”，事件记录回答“经过了哪些步骤才变成现在这样”。

### 13.6 结构化数据与 Markdown 正文分开

MySQL 保存结构化状态、关系和索引；文件系统保存完整 Markdown。这样既方便程序查询，也方便人
直接阅读课程和报告。

### 13.7 法条和 Artifact 使用关联表

使用独立关联表能够正确表达多对多关系，并通过外键保证 Artifact 和引用真实存在。

---

## 14. 总结

这套数据库围绕一条完整的学习闭环设计：

> 学员提交问卷 → 建立画像 → 生成学习路径 → 生成课程和题目 → 学员作答 → 更新 BKT →
> 生成反馈 → 更新画像。

MySQL 保存闭环中的结构化业务事实和关系；`artifacts/` 保存完整 Markdown；Milvus 保存法律资料
向量；后端静态 JSON 保存课程知识结构；`StateDict` 负责工作流运行时的数据传递。

数据库中的核心关系可以概括为：

> 学员拥有会话和画像，会话产生路径、轮次、题目和 Artifact，学员对题目作答，作答更新掌握度，
> 掌握度和反馈再影响下一次课程。
