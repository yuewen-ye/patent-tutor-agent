# 专利导学系统数据库设计讲解稿

> 适用场景：项目汇报、答辩、技术交流
> 讲解对象：了解软件系统，但不一定熟悉数据库、Agent 或 LangGraph
> 讲解目标：说明数据库为什么存在、保存什么、在业务流程中什么时候读写，以及当前实现边界
> 技术细节：以 `docs/patent-tutor-rdb-design.md` 和实际代码为准

---

## 1. 这份讲解稿是否仍然适用

整体讲解思路仍然适用，但需要使用当前口径：

- 生产业务数据库是 MySQL，不是 SQLite；
- 生产环境只有一个业务数据库，不再维护两个互相竞争的权威数据库；
- 工作流节点通过 `StateDict` 协作，MySQL 不是 Agent 之间的共享黑板；
- 题目由课程生成大模型动态产生，不是来自固定题库；
- 当前客观题由后端根据隐藏参考答案判题，不是反馈大模型判题；
- 反馈 Agent 使用已经得到的正误结果生成反馈和更新画像；
- 自动化脚本只是模拟学员作答，用于验证接口和数据库链路；
- 主观题的大模型语义评分、学员安全题目接口、持久化 checkpoint 和完整认证仍未实现。

讲解时不要把预留表说成已经完成的功能，也不要把自动化测试说成真实学员完成了答题。

## 2. 先用一句话讲清楚

> MySQL 是系统的业务账本，保存学员、会话、画像、学习路径、模型生成的题目、学员作答、
> 掌握度和审计记录；适合人阅读的 Markdown 正文保存在 `artifacts/`，数据库保存文件索引、
> 哈希和归属。

如果听众继续追问，可以补充：

> 工作流运行时通过 `StateDict` 传递节点状态，MySQL 负责把需要长期保存、查询和解释的数据
> 持久化。Milvus 负责法律资料向量检索，后端 JSON 负责静态知识 DAG 和易混淆对。

## 3. 30 秒讲解版本

> 系统使用一个 MySQL 数据库保存结构化业务数据。学员提交问卷后，系统创建课程会话并保存
> 问卷；诊断阶段读取历史画像，产生当前画像；Planner 读取画像和 BKT 掌握度，形成个性化
> 学习路径。Expert A、Expert B 和 Judge 完成课程生成后，模型生成的题目、内部参考答案、
> 会话状态和过程产物索引进入数据库。课程生成到这里暂停，等待真实学员学习并提交答案。
> 当前后端根据隐藏参考答案判题，并在同一事务中保存作答、更新掌握度、记录 BKT 变化，之后
> 创建独立反馈会话，由反馈 Agent 根据题目、回答和正误结果生成反馈并更新画像。Markdown
> 正文仍保存在 artifacts 中，MySQL 负责查询、关联和审计。

## 4. 五类数据分别放在哪里

| 数据 | 存放位置 | 讲解重点 |
|---|---|---|
| 学员、会话、画像、路径、题目、作答、BKT | MySQL | 需要事务、外键、唯一约束、查询和审计 |
| 当前完整工作流状态 | `session_states.state_json` | 用于会话详情和服务重启后的历史读取 |
| 课程、专家过程和反馈 Markdown | `artifacts/sessions/{session_id}/` | 适合人阅读和人工检查 |
| 法律语料和向量 | Milvus | 用于向量相似度检索 |
| 知识 DAG 和易混淆对 | `backend/app/curriculum/data/` | 全体学员共享的静态课程资产 |

需要强调：

- 数据库并不是只保存文件路径；
- 只有完整 Markdown 大正文主要保存在文件系统；
- 题干、参考答案、学员回答、画像和 BKT 等结构化业务数据都在 MySQL；
- `session_states` 是业务状态快照，不等于 LangGraph checkpoint。

可以用下面的比喻：

| 组件 | 比喻 |
|---|---|
| MySQL | 系统账本 |
| `artifacts/` | 课程档案柜 |
| Milvus | 法律资料检索库 |
| 后端 JSON | 课程知识地图 |
| `StateDict` | 当前工作台 |

## 5. 一次完整业务流程中数据库如何工作

### 5.1 获取问卷

前端请求 onboarding 问卷。

这一阶段：

- 问卷直接来自后端版本化 Markdown 定义；
- 不读取 MySQL；
- 不写入 MySQL。

### 5.2 学员提交问卷

系统主要写入：

| 数据 | 作用 |
|---|---|
| 学员记录 | 确保 learner ID 对应的学员存在 |
| 课程会话 | 创建 `teach` 会话，初始状态为 `running` |
| 原始问卷回答 | 保存题号、答案和问卷版本 |
| 完整诊断上下文 | 保存题目正文、选项、所选项和开放回答 |
| 初始状态和事件 | 支持后续会话查询和执行追踪 |
| 问卷 Artifact 索引 | 关联问卷和提交内容的 Markdown 文件 |

这里应特别说明：

> 原始回答用于审计；补充了题目正文和选项的上下文用于诊断 Agent。诊断 Agent 不是只看到
> “Q1 选 B”，而是能够同时看到题目、选项和学员选择。

### 5.3 初始诊断

诊断阶段读取：

- 本次问卷的完整上下文；
- 该学员已有的画像或历史记忆；
- 后端静态知识 DAG。

诊断完成后保存：

- 当前画像；
- 历史画像快照；
- 薄弱点投影；
- 会话状态和节点事件；
- 画像 Markdown 的 Artifact 索引。

讲解重点：

> 当前画像用于快速查询，历史画像用于解释学员是怎样变化的。完整画像中可以包含所有知识
> 节点的先验状态，但只有真正发生观测或更新的节点才需要进入 mastery 当前值表。

### 5.4 Planner 生成学习路径

Planner 读取：

- 当前学员画像；
- 已有观测的知识点 BKT 掌握度；
- 静态知识 DAG 和易混淆对。

Planner 保存：

- 带版本的学习路径；
- 出题范围和教学迭代指令；
- 会话状态和事件；
- 路径与双轴快照的 Artifact 索引。

这一阶段证明 MySQL 不只是给前端使用。数据库中的画像和 mastery 会影响后续教学路径。

### 5.5 专家协作与 Judge

Expert A、Expert B 和 Judge 主要通过当前 `StateDict` 协作，不会把数据库当成 Agent 间消息总线。

每个节点完成后，持久化层持续保存：

| 数据 | 作用 |
|---|---|
| 会话状态 | 保存当前完整结构化结果 |
| 节点事件 | 记录执行阶段和结果 |
| 专家轮次 | 记录轮次、整合次数和 Judge 决策 |
| Artifact 索引 | 关联草稿、互评、修订、课程包和 Judge 报告 |
| 模型生成题目 | 保存题干、选项、内部参考答案和知识点 |
| 法条引用 | 保存引用来源及其与 Artifact 的关系 |

Judge 要求修改时，系统增加新的整合尝试，不覆盖之前的课程和审核记录。

### 5.6 课程完成后暂停

Judge 通过后，课程会话标记为 `completed`。

这时真实产品不应该立即自动提交练习，而应该：

1. 前端向学员展示课程；
2. 前端只展示题干、选项和必要元数据；
3. 隐藏内部参考答案和评分证据；
4. 等待真实学员完成学习和作答；
5. 学员稍后提交原始答案。

课程生成和学员作答之间可以间隔几分钟、几小时甚至几天。

### 5.7 学员提交练习

系统先检查：

- 原课程会话是否存在；
- 课程是否属于该学员；
- 课程是否已经完成；
- 题目是否属于该课程；
- 幂等键是否已经提交过。

随后读取：

| 数据 | 目的 |
|---|---|
| 题目 | 获取内部参考答案和知识节点 |
| 已有作答 | 防止重复提交 |
| 当前 mastery | 计算更新前的 P(L) |

当前后端判题规则是：

- 有参考答案时进行规范化精确比较；
- 得不到可靠二值结果时允许标记为未判定；
- 客户端提交的“我答对了”不能覆盖服务端结果。

判题后在同一事务中保存：

| 数据 | 内容 |
|---|---|
| 作答记录 | 原始答案、判题状态、结果、来源和幂等键 |
| 当前 mastery | 新 P(L)、观测次数和正确/错误次数 |
| mastery 事件 | 更新前值、后验值、最终值和 BKT 参数 |

然后创建独立的 `feedback` 会话，通过 `parent_session_id` 指向原课程会话。

### 5.8 反馈阶段

反馈 Agent 接收：

- 题目 ID；
- 学员原始回答；
- 后端已经确定的正误结果；
- 当前画像；
- 课程和 Judge 上下文。

当前反馈 Agent 不独立判题。它主要负责：

- 解释学员表现；
- 生成下一步建议；
- 更新画像提示和薄弱点；
- 形成反馈报告。

反馈完成后保存：

- 最新画像；
- 新画像历史快照；
- 最新薄弱点；
- 反馈日志；
- 反馈会话状态和事件；
- 反馈报告等 Artifact 索引。

至此完成“问卷—诊断—规划—教学—练习—BKT—反馈”闭环。

## 6. 自动化脚本和真实学员流程的区别

端到端脚本可以调用所有接口，但它只是技术测试。

在 `correct` 模式下，脚本会从内部完整课程状态中读取模型生成的参考答案，并把它作为模拟学员
答案提交；在 `incorrect` 模式下提交固定错误值。

它能证明：

- 大模型成功生成题目；
- 题目成功写入数据库；
- 作答接口能够接收数据；
- 后端能够判题；
- BKT 和反馈链路能够完成；
- MySQL 和 Artifact 能够查询。

它不能证明：

- 真实学员阅读了课程；
- 真实学员理解了题目；
- 真实学员独立完成了作答；
- 当前系统已经具备主观题语义评分能力。

汇报时应称它为“真实服务和真实模型的端到端链路验证”，不能称为“真实学员测试”。

## 7. 数据表如何分组讲解

当前共有 25 张表。汇报时不要逐表背字段，按八组讲即可：

| 分组 | 主要表 | 讲解重点 |
|---|---|---|
| 数据库版本 | `schema_migrations` | 记录哪些结构升级已经执行 |
| 学员与认证 | `students`、`auth_sessions` | 学员身份已使用，完整认证仍未实现 |
| 会话运行 | `sessions`、`session_states`、`session_events`、`session_checkpoints`、`rounds` | 保存会话、状态、事件、轮次和预留 checkpoint |
| 学员画像 | `student_profiles`、`profile_history`、`student_weak_points`、`memory_items` | 当前画像、历史画像、薄弱点和兼容记忆 |
| 自适应学习 | `student_node_mastery`、`mastery_events`、`learning_paths`、`session_directives` | 当前 BKT、变化审计和路径规划 |
| 教学闭环 | `onboarding_responses`、`questions`、`attempts`、`feedback_logs` | 问卷、模型生成题目、作答和反馈 |
| 产物审计 | `artifacts`、`legal_citations`、`artifact_citations` | 文件索引、哈希和法条关联 |
| 静态目录预留 | `knowledge_nodes`、`confusion_pairs` | 表已存在，当前运行时仍读取后端 JSON |

如果被问“多出来的表是否都有必要”，可以回答：

- 当前主链路必要：`schema_migrations`、`memory_items`、`session_states`、`session_events`、
  `mastery_events`、`artifact_citations`；
- 当前只是预留：`auth_sessions`、`session_checkpoints`。

## 8. 最重要的实体关系

建议按下面的顺序口头讲：

1. 一个学员可以有多个课程和反馈会话；
2. 每个学员只有一份当前画像，但有多份历史画像；
3. 每个学员在每个知识点上只有一个当前 mastery；
4. 一个课程会话可以产生多个专家轮次、学习路径、题目和 Artifact；
5. 题目属于课程会话，作答属于反馈会话；
6. 作答通过题目找到原课程，通过反馈会话找到本次反馈；
7. 一次已判定作答最多产生一次 mastery 更新事件；
8. 反馈会话通过父会话关系指向原课程会话。

最重要的一条链路是：

> 学员 → 课程会话 → 模型生成题目 → 学员作答 → BKT 更新事件 → 反馈会话 → 新画像。

## 9. 七个核心设计理由

### 9.1 只使用一个生产业务数据库

画像、掌握度和会话都进入同一个 MySQL schema，避免多个数据库同时声称自己是最新数据。
SQLite 只保留为单元测试替身。

### 9.2 `StateDict` 和数据库职责分开

`StateDict` 负责一次工作流内部的即时协作；MySQL 负责长期保存、查询和审计。数据库不是 Agent
之间传递消息的黑板。

### 9.3 当前投影和历史记录分开

| 当前数据 | 历史解释 |
|---|---|
| 当前画像 | 画像历史 |
| 当前 mastery | 作答和 mastery 事件 |
| 当前会话状态 | 节点事件和专家轮次 |

当前表便于快速查询，历史表用于回答“为什么发生变化”。

### 9.4 判题结果不能由客户端决定

题目内部参考答案保存在服务端。前端提交原始答案，后端决定判题状态，避免客户端伪造
`is_correct=true`。

### 9.5 作答和 BKT 更新使用同一事务

作答、当前 mastery 和 mastery 审计要么一起成功，要么一起回滚，避免半完成状态。

### 9.6 结构化状态和 Markdown 分开

MySQL 保存机器容易处理的 JSON、关系和索引；Artifact 文件保存人容易阅读的 Markdown。
数据库通过路径、会话归属和 SHA-256 把两者连接起来。

### 9.7 预留表不等于功能已实现

存在 `auth_sessions` 不代表登录已经完成；存在 `session_checkpoints` 也不代表工作流已经能够
中断续跑。汇报必须区分表结构、实际读写和完整产品能力。

## 10. 当前判题设计应该怎样讲

推荐表述：

> 课程生成大模型负责生成题目、参考答案和必要证据；学员提交原始答案后，当前后端对固定答案
> 题进行确定性判定，再把题目、回答和判题结果交给反馈 Agent。反馈 Agent 负责解释表现和更新
> 画像，不负责重新判定客观题。

不要表述为：

> 所有题目都由反馈大模型直接判断正误。

当前适合：

- 单选题；
- 判断题；
- 固定枚举答案；
- 可以规范化比较的短答案。

当前不适合：

- 开放式简答；
- 案例分析；
- 权利要求撰写；
- 存在多种合理表达的主观题。

后续主观题评分需要增加：

- 评分方式；
- 评分量表及版本；
- 分项得分和总分；
- 大模型评分理由与证据；
- 评分模型和提示词版本；
- 人工复核状态；
- 最终确认结果。

这些目前是演进方向，不是已经实现的能力。

## 11. 前端如何使用数据库数据

前端不会直接连接 MySQL，而是通过 FastAPI：

| 前端需求 | 使用方式 |
|---|---|
| 查看会话列表 | 会话列表接口读取 MySQL 摘要 |
| 查看会话状态 | 会话详情接口读取活动状态或持久化快照 |
| 展示结构化课程 | 使用课程的结构化输出 |
| 阅读 Markdown | 调用 Artifact API，由后端校验路径后返回正文 |
| 查看画像和 mastery | 调用学员画像、历史和掌握度接口 |
| 接收实时事件 | 使用 SSE/WebSocket；历史事件写入 MySQL |
| 提交练习 | 提交题目 ID、原始答案、作答耗时和幂等键 |

需要特别指出：

> 当前完整会话状态可能包含内部参考答案，因此不能直接作为正式学员题目接口。正式前端还需要
> 学员安全题目 DTO，只返回题干、选项和必要元数据。

前端展示 Markdown 时，也不需要先查询路径再直接读取磁盘。Artifact API 会完成数据库归属查询、
路径校验和正文读取。

## 12. 当前实现边界

### 已实现

- MySQL 连接池和版本化迁移；
- 会话、完整状态快照、节点事件和专家轮次；
- 原始问卷及包含题目、选项的诊断上下文；
- 当前画像、历史画像和薄弱点；
- 当前 mastery 和 BKT 状态转移审计；
- 学习路径和教学指令；
- 模型生成题目和内部参考答案；
- 固定答案的服务端确定性判题；
- 幂等作答及作答、BKT 同事务更新；
- 独立反馈会话和画像更新；
- Artifact 索引、路径和 SHA-256；
- 法条和 Artifact 的多对多关联。

### 尚未实现

- 主观题 LLM 语义评分；
- 学员安全题目读取接口；
- MySQL 持久化 LangGraph checkpoint；
- 注册、登录和令牌撤销等完整认证；
- 静态知识 DAG 和易混淆对的数据库 seed 运行时切换；
- 法条自动核验；
- Artifact 缺失文件或索引的自动修复。

正确结论是：

> 数据库主业务闭环已经实现，但主观题评分、答案隔离、工作流中断恢复和完整认证仍需继续建设。

## 13. 常见问题与简短回答

### 数据库里是不是全部都是 Artifact 路径？

不是。数据库还保存学员、画像、会话、状态、路径、题目、参考答案、学员作答、BKT 和审计记录。
只有完整 Markdown 大正文主要保存在文件系统。

### 工作流运行时真的会读取数据库吗？

会。诊断阶段读取历史画像，Planner 读取当前画像和 mastery，学员作答时读取题目和当前 mastery。
专家生成阶段主要使用当前 `StateDict`，不会每一步重新查询数据库。

### MySQL 是不是只为前端服务？

不是。它同时服务于运行时个性化决策、服务端判题、BKT 更新、历史查询和审计。

### 题目来自题库吗？

不是固定题库。题目由课程生成大模型根据学员画像、学习路径和知识点动态生成，之后作为题目实例
保存到数据库。

### 学员答案是谁判定的？

当前固定答案题由后端根据隐藏参考答案判定。反馈 Agent 使用判题结果生成解释和更新画像，不是
当前的判题者。

### 自动化脚本为什么能完成整个流程？

因为它模拟了学员。在正确模式下直接使用内部参考答案，在错误模式下提交固定错误值。它验证的是
技术链路，不代表真实学员完成了学习。

### 数据库能让中断的工作流继续执行吗？

目前不能。状态快照支持重启后查询，但真正中断续跑需要持久化 LangGraph checkpoint。

### `schema_migrations` 保存什么？

它只记录哪些数据库结构升级已经执行以及执行时间，不保存业务数据，也不保存完整 SQL。

### 为什么还需要 `mastery_events`？

当前 mastery 只能说明现在是多少；mastery 事件能够解释它由哪次作答、使用什么 BKT 参数、从
多少变化到多少。

### 为什么不把 Markdown 全放进 MySQL？

长 Markdown 更适合文件读取、下载和人工检查；MySQL 更适合结构化数据、关系、事务和查询。

### 为什么还需要 Milvus？

MySQL 负责精确关系查询，Milvus 负责法律资料向量相似度检索，两者解决的问题不同。

## 14. 推荐的 5 分钟讲解顺序

### 第 1 分钟：系统定位

讲清楚：

- MySQL 是业务账本；
- Artifact 是课程档案；
- Milvus 是法律资料检索库；
- `StateDict` 是工作流当前工作台。

### 第 2 分钟：问卷到课程

讲：

- 问卷提交后创建课程会话；
- 诊断生成当前画像和历史画像；
- Planner 读取画像和 mastery；
- 专家和 Judge 生成课程、题目和过程产物。

### 第 3 分钟：学员作答到反馈

讲：

- 课程完成后暂停等待真实学员；
- 后端接收原始回答并判定固定答案题；
- 作答、mastery 和 mastery 事件同事务；
- 创建独立反馈会话并更新画像。

### 第 4 分钟：数据库设计亮点

讲：

- 当前值与历史分离；
- 幂等和事务；
- 结构化状态与 Markdown 分离；
- 法条与 Artifact 使用多对多关系；
- 数据库不是 Agent 黑板。

### 第 5 分钟：实现边界

明确：

- 主观题 LLM 评分未实现；
- 学员安全题目接口未实现；
- checkpoint 和认证是预留能力；
- 自动化脚本只是端到端测试。

## 15. 可直接照着讲的 1 分钟版本

> 我们的数据库不是简单保存课程文件路径，而是整个个性化教学闭环的业务账本。学员提交问卷后，
> MySQL 会保存问卷、课程会话和初始状态；诊断阶段生成当前画像和历史画像，Planner 再读取画像
> 和知识点掌握度生成个性化路径。Expert A、Expert B 和 Judge 生成课程时，系统持续保存会话
> 状态、节点事件、专家轮次、模型生成的题目和 Artifact 索引。课程完成后系统会暂停，等待真实
> 学员作答。当前固定答案题由后端根据隐藏参考答案判定，并在同一事务中保存作答、更新 BKT、
> 记录掌握度变化，之后创建独立反馈会话，由反馈 Agent 生成解释并更新画像。完整 Markdown 放在
> artifacts 中，MySQL 保存结构化数据、路径、哈希和归属；Milvus 单独负责法律资料向量检索。
> 当前数据库主闭环已经实现，但主观题大模型评分、学员答案隔离接口、持久化 checkpoint 和完整
> 认证仍是后续工作。

## 16. 讲解时不要说错的四句话

不要说：

1. “数据库里保存的全部都是 Artifact 路径。”
2. “数据库是 Agent 之间交换消息的共享黑板。”
3. “反馈 Agent 已经负责判断所有题目的正误。”
4. “自动化脚本跑通就代表真实学员完成了学习。”

应该说：

1. 数据库保存结构化业务事实，Artifact 表只负责文件索引；
2. Agent 通过 `StateDict` 协作，数据库负责持久化和审计；
3. 当前固定答案题由后端判定，反馈 Agent 负责解释和画像更新；
4. 自动化脚本验证真实接口、模型和 MySQL 链路，但学员行为是模拟的。
