## 1. 向量数据库选型
### 1.1 常见方案对比

[Milvus_Lite]：Milvus Lite 是 Milvus 的轻量级嵌入式版本，支持单文件运行，适应开发测试和小规模部署场景；支持多种向量索引和布尔过滤，适配复杂检索需求；支持 BM25 + 向量混合检索，适配专利文档多维度检索场景；可无缝迁移到 Milvus 独立服务或云托管版，适配项目未来扩展需求。

[FAISS]：FAISS 是 Facebook 开源的向量检索库，性能优异但功能单一，适应高吞吐量检索场景；仅支持向量检索，不适配混合检索需求；部署简单但生态较弱，不适配复杂业务场景；不支持平滑升级，不适配长期演进项目。

[Chroma]：Chroma 是专为 LLM 应用设计的向量数据库，API 简洁易用，适应快速开发场景；功能较为基础，不适配复杂检索需求；支持嵌入式部署，适配小规模知识库场景；生态较新，不适配企业级应用需求。

[Pinecone]：Pinecone 是云原生向量数据库服务，提供全托管服务，适应大规模生产场景；功能全面但依赖云端，不适配离线部署需求；按使用量付费，不适配成本敏感项目；闭源服务，不适配自主可控要求。

### 1.2 [Milvus_Lite] 选择理由

Milvus Lite 是支持嵌入式部署的轻量级向量数据库，单文件运行无需独立服务进程，适配项目开发测试和初期部署场景；支持多种向量索引和布尔过滤，满足复杂检索需求；支持 BM25 + 向量混合检索，适配专利文档多维度检索场景；可无缝迁移到 Milvus 独立服务或云托管版，适配项目未来扩展需求；Python SDK 完善且与 LangChain 深度集成，适配项目技术栈。

### 1.3 部署方式

Milvus Lite 采用单文件嵌入式部署方式，通过 Python 包即可集成，无需额外服务进程，依赖声明为 pymilvus 包（版本要求 2.4.0 及以上），初始化时指定本地存储路径即可自动创建向量集合，支持 CPU 和 GPU 加速适配不同硬件环境；未来可无缝迁移到 Milvus Standalone 独立服务或 Milvus Cluster 分布式集群，也可选择 Milvus Cloud 云托管服务，数据和 API 完全兼容无需修改代码，形成开发测试到生产环境的平滑升级路径。

## 2. Embedding模型[BGE-M3]选型理由
### 2.1 常规模型对比

[BGE-M3]：BGE-M3 是支持多粒度嵌入的开源模型，支持短/长/超长文本处理，适应中英文混合专利文档场景；具有 1024 维向量维度，向量信息较为丰富，语义区分度更高，模型适合专利文档的精确语义检索需求；支持中英双语，适配专利文档中英文混合场景；采用 MIT 开源许可，是开源模型，适配项目商业部署需求。

[text-embedding-3]：text-embedding-3 是 OpenAI 推出的闭源嵌入模型，具有 1536 维向量维度，向量信息最丰富；支持多语言但中文优化不足，不适配中文专利文档场景；闭源商用需付费，不适配项目开源部署策略。

[m3e-base]：m3e-base 是专门针对中文优化的嵌入模型，在中文文本上经过深度训练，适应纯中文专利文档场景；具有 768 维向量维度，向量更紧凑，计算更快，但可能丢失一些细微的语义差异，模型适合轻量级语义检索（响应快、高频查询）；采用 Apache-2.0 许可，适配项目商业集成需求。

## 3. 切片策略对比【切片策略】

### 3.1 策略对比

[语义切片]：语义切片基于语义边界进行切割，保持句子和段落的完整性，适应法律、技术文档；语义完整性好，避免上下文断裂，适配专利法条等专业文档；实现复杂度较高，需要 NLP 模型支持，适配对语义准确性要求高的场景；处理速度相对较慢，适配中小规模知识库场景。

[固定长度切片]：固定长度切片按固定字符数均匀切割，实现简单速度快，适应普通文本场景；可能切断上下文导致语义断裂，不适配复杂专业文档；实现简单无需额外 NLP 依赖，适配快速开发场景；处理速度快，适配合大规模知识库场景。

### 3.2 混合策略设计

混合策略以语义切片为主、固定长度切片兜底，首先进行分句处理，当句子长度小于阈值时采用语义切片保持完整性；当句子长度大于等于阈值时采用固定长度切片并使用重叠窗口，兼顾语义完整性和处理效率；适配专利文档中既有完整法条又有超长技术描述的混合场景。
